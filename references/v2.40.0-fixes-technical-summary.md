מסמך זה מתעד בדיוק מה הקוד עושה, לפי שם קובץ ופונקציה, לא תיאור כללי.
קובץ: `extension/background.js`, פונקציה: `publishPost(post, token, expectedUser)`.

## 1. תמונה — `DOM.setFileInputFiles` ממוקד לדיאלוג

**מיקום:** `background.js`, בתוך `publishPost()`, שורות 181–240 (בלוק "2. תמונה").
פונקציית עזר: `downloadToTempFile(url)` (שורות 44–74).

**מה הקוד עושה בפועל, צעד-צעד:**

1. `downloadToTempFile(imageUrl)` — קורא ל-`chrome.downloads.download({url, filename, saveAs:false})`
   ומחכה לאירוע `chrome.downloads.onChanged` עם `state.current==="complete"`, ואז
   `chrome.downloads.search({id})` כדי לקבל את ה-path המקומי המלא בדיסק.
   **למה:** `DOM.setFileInputFiles` (CDP) דורש path מקומי אמיתי — לא URL, לא Blob.
   מקור התמונה הוא URL מרוחק (מאוחסן בענן), אז אין ברירה אלא להוריד קודם.

2. `sendDebuggerCommand(tabId, "DOM.enable")` ואז `"DOM.getDocument", {depth:-1, pierce:true}`
   (שורות 194–197) — מפעילים את מעקב ה-DOM של CDP. נדרש לפני שאפשר לקרוא ל-`DOM.requestNode`.

3. **הליבה של התיקון** (שורות 199–219): במקום לחפש `input[type="file"]` בכל
   ה-`document` (מה שעשינו בגרסאות קודמות ותפס input שגוי — פייסבוק מחזיקה כמה
   `input[type=file]` נסתרים בעמוד: תמונת פרופיל, תמונת קבוצה וכו'), עכשיו:
   - `sendDebuggerCommand(tabId, "Runtime.evaluate", {expression: "...", returnByValue:false})`
     מריץ JS בעמוד שמוצא את ה-`[role="dialog"]` שמכיל `[role="textbox"]`
     (זה חלון הכתיבה הספציפי, לא כל דיאלוג אחר בעמוד) ומחזיר `objectId` שלו
     (כי `returnByValue:false` — מקבלים reference, לא ערך).
   - `sendDebuggerCommand(tabId, "DOM.requestNode", {objectId})` — ממיר את
     ה-`objectId` (רפרנס JS) ל-`nodeId` (מזהה CDP/DOM).
   - `sendDebuggerCommand(tabId, "DOM.querySelector", {nodeId: dialogNodeId, selector:'input[type="file"]'})`
     — מחפש את ה-input **בתוך אותו nodeId בלבד**, כלומר בתוך חלון הכתיבה, לא בכל הדף.

4. `sendDebuggerCommand(tabId, "DOM.setFileInputFiles", {files:[filename], nodeId})`
   (שורה 231) — הפקודה שבפועל "מזריקה" את הקובץ לתוך ה-`<input>`, כאילו המשתמש
   בחר אותו דרך בורר קבצים — כולל אירוע `change` אמיתי שReact/פייסבוק קולטים.
   **בלי לפתוח שום דיאלוג, בלי ללחוץ על שום כפתור נראה.**

5. אימות בפועל (לא רק שהפקודה לא זרקה שגיאה): `chrome.scripting.executeScript`
   בודק `dialog.querySelector('img[src^="blob:"]')` — `blob:` הוא URL מקומי שנוצר
   ע"י הדפדפן ברגע שקובץ מקומי מוצג כתצוגה מקדימה, לכן זה סימן אמין. (בגרסה
   קודמת בדקנו `img[src*="scontent"]` — זה false positive כי גם תמונת פרופיל
   קיימת מוגשת מ-scontent).

6. ניקוי (finally, שורות ~245): `chrome.downloads.removeFile(downloadId)` +
   `chrome.downloads.erase({id})` — מוחקים את הקובץ הזמני מהדיסק.

## 2. קישור וואטסאפ — טקסט רגיל, בלי מניפולציה על הכרטיס

**מיקום:** `background.js`, בלוק "4. הכנס טקסט מלא", שורה 273 בערך.

**מה הקוד עושה:**
```js
if (post.campaign.whatsappLink) fullText += `\n\n📱 ליצירת קשר בוואטסאפ: ${post.campaign.whatsappLink}`;
```
הקישור (למשל `https://wa.me/972...`) מצורף כטקסט פשוט לתוך `fullText`, ומוכנס
כולו יחד (תוכן + קישור) דרך `chrome.debugger.sendCommand({tabId}, "Input.insertText", {text: fullText})`.

**מה השתנה מגרסה קודמת:** בגרסאות 2.26–2.38 היה קוד נוסף (שנמחק לגמרי עכשיו)
שניסה לאתר את כרטיס התצוגה המקדימה (`WA.ME` preview card) שפייסבוק יוצרת
אוטומטית כשהיא מזהה URL, ולמצוא וללחוץ על כפתור ה-X שלו כדי "להסיר את הפס
האפור". זה היה שביר (הסלקטורים לא מצאו את הכפתור הנכון בעקביות) וגרם לבעיות
נוספות. **התיקון: לא לגעת בכרטיס בכלל.** פייסבוק הופכת אוטומטית טקסט שהוא URL
לקישור כחול לחיץ + כרטיס תצוגה — וזה בסדר גמור להשאיר כמו שזה.

## 3. סגירת טאב — מאזין ל-`Page.javascriptDialogOpening`

**מיקום:** `background.js`, תחילת `publishPost()`:
- הצהרת המשתנה: שורה 121 (`let dialogListener = null`, מחוץ ל-`try` כדי שה-`finally` יוכל לגשת אליו)
- הגדרת המאזין: שורות 131–139
- הסרתו: שורה 351 (בתוך `finally`, אחרי הניווט אבל לפני `detach`)

**מה הקוד עושה:**
```js
dialogListener = (source, method) => {
  if (source.tabId !== tabId) return;
  if (method === "Page.javascriptDialogOpening") {
    chrome.debugger.sendCommand({ tabId }, "Page.handleJavaScriptDialog", { accept: true });
  }
};
chrome.debugger.onEvent.addListener(dialogListener);
```
מיד אחרי `chrome.debugger.attach`, רושמים מאזין גלובלי לאירועי CDP
(`chrome.debugger.onEvent`). כשפייסבוק מנסה לפתוח דיאלוג `beforeunload`
("האם לעזוב את האתר?") — CDP שולח אירוע `Page.javascriptDialogOpening` **לפני**
שהדיאלוג בפועל "תפוס" וממתין לקלט אנושי. המאזין מיד שולח בחזרה
`Page.handleJavaScriptDialog({accept:true})` — כאילו המשתמש לחץ "עזוב" —
והדיאלוג נדחה אוטומטית, בלי שום המתנה.

**מה השתנה מגרסה קודמת:** בגרסאות קודמות ניסינו "לעקוף" את הדיאלוג ע"י ניווט
ל-`about:blank` דרך CDP *בזמן שה-debugger מחובר* (בהנחה ש-CDP מדלג על
beforeunload). זה עבד רק חלקית ובאופן לא עקבי. **התיקון: לא לנסות לעקוף אחרי
שהדיאלוג כבר עלול להיפתח — להאזין ולדחות אותו באופן פרואקטיבי ברגע שהוא נפתח,**
לפני שהוא מספיק "להיתקע". ה-`Page.navigate({url:"about:blank"})` ב-`finally`
עדיין קיים כפעולת הניווט עצמה, אבל עכשיו יש רשת ביטחון (המאזין) שתופסת כל
דיאלוג שבכל זאת ינסה להיפתח כתוצאה מהניווט.
